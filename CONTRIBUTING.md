# Contributing to Yardstick

The repo's main asset is the benchmark table in [`RESULTS.md`](RESULTS.md). **Every contribution is a new row, a new column, or a new pivot of that table.** This file is the step-by-step for getting a row in.

If you want the 30-second version, read the [Contributing a row](README.md#-contributing-a-row) block in the README and come back here for the gotchas.

## The three ways to add a row

| Kind of row | What it lets readers compare | What you supply |
|---|---|---|
| **New device** | The same runtime + model running on your hardware vs an existing row's hardware | A page under `devices/<your-device>.md` + JSONL files in `results/raw/` |
| **New model** | A bigger / smaller / differently-quantized model on existing runtime+device | An entry in `ios/BenchmarkApp/Sources/Models/ModelCatalog.swift` + JSONL |
| **New runtime** | A new backend (e.g. a different llama.cpp build, a new ExecuTorch variant) against an existing model+device | A new adapter file under `ios/BenchmarkApp/Sources/Runtimes/` conforming to `LLMRuntime` + JSONL |

All three end the same way: drop JSONL into `results/raw/`, run `python scripts/render_results.py`, commit.

## 0. One-time setup

Mac CLI:

```sh
git clone https://github.com/john-rocky/Yardstick.git
cd Yardstick/ios/BenchmarkApp
xcodegen generate                              # regenerate Xcode project from project.yml
xcodebuild -project BenchmarkApp.xcodeproj \
           -scheme yardstick \
           -destination "platform=macOS" \
           -configuration Release build

# Symlink the binary so commands below stay readable:
ln -sf ~/Library/Developer/Xcode/DerivedData/BenchmarkApp-*/Build/Products/Release/yardstick ../../yardstick
cd ../..
./yardstick list
```

iPhone app:

```sh
cd ios/BenchmarkApp
./scripts/bootstrap.sh           # downloads llama.xcframework + Anemll source
open BenchmarkApp.xcodeproj
# - Set your Team in Signing & Capabilities
# - Pick your iPhone as the run destination
# - ⌘R
```

Both builds work without changes once `xcodegen` is on `PATH` (`brew install xcodegen`).

## 1. Run the benchmark

The CLI version is identical on Mac:

```sh
./yardstick run --task short-chat \
                --runtime mlx-swift \
                --model mlx-community/gemma-4-e2b-it-4bit \
                --output results/raw/<device>-mlx-gemma-4-e2b-short-chat-run1.jsonl
```

Three things matter:

- **Run 3 times** for the same `(runtime, model, task, device)` tuple. The renderer uses median for `n ≥ 1` but the "trustworthy" sections of `RESULTS.md` cite `n ≥ 3`.
- **Filename convention** is `<device>-<rest>.jsonl`. The renderer treats the *first* dash-separated segment as the device label and looks it up in `scripts/render_results.py::DEVICE_DISPLAY`. The rest of the filename is for humans; the renderer reads the runtime / model / task fields from the JSONL itself.
- **Use the same prompt + settings** for cross-runtime apples-to-apples. The CLI bakes the Task-A prompt in; just match the task name.

For the iPhone app, run the tasks via the app's UI, export the result JSONLs (top-right "Export" sheet), and AirDrop them back to the Mac into `results/raw/`.

## 2. Add your device to `DEVICE_DISPLAY`

If your device label isn't already mapped in `scripts/render_results.py::DEVICE_DISPLAY`, add it:

```python
DEVICE_DISPLAY = {
    "m4max": "Mac M4 Max",
    "iphone17pro": "iPhone 17 Pro",
    "ipadprom4": "iPad Pro M4",
    "<your-label>": "<Pretty Name>",
    ...
}
```

And add `devices/<your-label>.md` describing the hardware / OS / build / signing.

## 3. Regenerate the tables

```sh
python scripts/render_results.py
```

This rewrites the block in `RESULTS.md` between

```
<!-- BEGIN: generated by scripts/render_results.py -->
<!-- END: generated by scripts/render_results.py -->
```

The hand-written intro, methodology, and "Failed runs" sections survive.

To check without writing (useful in pre-commit):

```sh
python scripts/render_results.py --check
```

Returns non-zero and prints a diff if the table is stale.

## 4. Open the PR

Put **JSONLs + RESULTS.md** in the same PR. CI runs `--check` and fails if they're out of sync.

PR title pattern: `<device> <runtime>: add Task A rows for <model-list>`.

## Failed runs are signal — please record them

If a run produces an error rather than a result, add a row to `RESULTS.md`'s "Failed runs" section by hand. We keep them visible so the next person debugging the same combination doesn't repeat the work. If the failure is upstream of Yardstick (a backend SDK, a downloader, an HF repo layout), link to the issue / PR you filed against the upstream project.

## What constitutes a *good* row

- **A model size the existing tables don't cover.** A second runtime for a model we already have is gold; a third device for a row we already have is gold.
- **`n = 3` median.** Single-shot rows are accepted (the renderer marks `n = 1`) but the "trustworthy" sections of `RESULTS.md` only cite `n ≥ 3`.
- **The exact same prompt and decode settings as everything else** so cross-runtime cells are apples-to-apples.
- **Thermal state recorded.** If your device threw a `serious` or `critical` state, that's not a failed run — that's the data point.

## What NOT to do

- **Don't hand-edit the rendered tables.** They're auto-generated, and CI will overwrite your edits on the next contributor's PR. Edit JSONL, run the script, commit both.
- **Don't average across devices or runtimes.** The harness exists to expose differences; collapsing them defeats the point.
- **Don't tune the prompt or decode settings per backend.** A faster number under different settings isn't comparable.
- **Don't ship anonymized device fingerprints.** Identifiable hardware (chip / RAM / OS / build) is what makes the row useful for the next reader.

## Questions

Open a GitHub issue, or reach out via the contact listed in [`README.md`](README.md). Genuine "I want to run this on my <device>" requests are the entire point of the project — please don't sit on them.
