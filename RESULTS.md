## Quality bar

| Discipline | What we do today |
|---|---|
| **Iterations** | `n ≥ 3` is preferred; tables show median for any group with `n ≥ 1`. The "Cross-runtime" cell at the top of "At-a-glance" is only trustworthy at `n ≥ 3`. |
| **Cold vs. warm** | The runtime is reloaded for every CLI invocation (no daemon), so every measurement is "runtime-cold". For `n ≥ 3` rows the first run also pays the cold-OS-cache penalty (downloads, kernel JIT, ANE compile). |
| **Prompt + decode** | Single fixed prompt (`"Explain what on-device AI means in simple terms."`), temperature 0.0, max 128 tokens. Identical across runtimes. |
| **Device state** | Plugged in, "High Power" mode, idle desktop, no concurrent benchmarks. Thermals stayed `nominal` for every Task A row to date. |

## Headline observations (read this after the tables)

These are the non-trivial facts the harness has surfaced so far. They are **not** general truths — they are conclusions valid for the rows above, on the devices listed, with the quants listed.

- **Runtime rank inverts with model size (Gemma 4 on M4 Max).** llama.cpp / Q4_K_M is ~2.1× faster than MLX-Swift / Q4 on Gemma 4 E2B (119.6 vs 56.6 tok/s), then ~0.90× on Gemma 4 E4B (40.65 vs 45.2 tok/s). Don't extrapolate runtime choice from one model size to another without measuring.
- **CoreML / ANE trades throughput for memory.** On Gemma 4 E2B, peak memory is 1052 MB on CoreML vs 2833 (MLX) / 3168 (llama.cpp) — a 2.7× / 3.0× reduction at the cost of ~half the decode throughput. LFM 2.5 350M on CoreML peaks at only 98 MB; ANE residency is unusually frugal.
- **First MLX-Swift run pays a kernel-JIT cost that subsequent runs do not.** Across the three runs of Gemma 4 E2B, the first invocation's TTFT (369 ms) and prefill (60.8 tok/s) are outliers — runs 2-3 settle at 84-100 ms TTFT and 306-348 tok/s prefill. If you have a single shot to load and measure, expect the cold number.

## How to add a row

The "in the trenches" version is in [`CONTRIBUTING.md`](CONTRIBUTING.md); the one-liner is:

1. Build the runner (Mac CLI or iPhone app — see CONTRIBUTING).
2. Run the task and save the JSONL under `results/raw/`:
   ```sh
   for run in 1 2 3; do
     yardstick run --task short-chat \
                   --runtime <id> \
                   --model <id-or-hf-repo> \
                   --output results/raw/<device>-<runtime>-<model>-short-chat-run${run}.jsonl
   done
   ```
   Filename convention: **first dash-separated segment = device label** (one of the keys in `scripts/render_results.py::DEVICE_DISPLAY`). The rest of the filename is for humans; the renderer reads runtime / model / task from the JSON itself.
3. Regenerate the tables:
   ```sh
   python scripts/render_results.py
   ```
4. PR the JSONLs **and the updated `RESULTS.md`** together. CI runs `--check` on every PR to make sure they stay in sync.
